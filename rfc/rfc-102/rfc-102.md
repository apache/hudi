# RFC-102: REGISTER_ONLY Bootstrap Mode

## Problem

Bootstrapping large tables into Hudi requires reading every partition — either extracting record keys (METADATA_ONLY) or fully rewriting data (FULL_RECORD). For cold partitions that will never receive updates, this is wasted work.

**REGISTER_ONLY** registers external files in Hudi without reading their contents: empty skeleton files + a mapping to the source. This enables fast onboarding of large historical datasets.

## Design

### Three-Tier Bootstrap

| Mode | Skeleton file | Meta columns | Record-level ops |
|---|---|---|---|
| FULL_RECORD | N/A (standalone Hudi file) | Populated | Yes |
| METADATA_ONLY | Contains meta columns | Populated | Yes |
| REGISTER_ONLY | Empty (0 rows) | NULL | No — promote first |

A new `BootstrapMode.REGISTER_ONLY` enum value is added. `BootstrapModeSelector` implementations (e.g. extending `BootstrapRegexModeSelector`) classify partitions into these three tiers.

### Timeline

| Instant | Purpose |
|---|---|
| `00000000000001` | METADATA_ONLY skeleton files + main bootstrap index |
| `00000000000002` | FULL_RECORD bulk-insert |
| `00000000000003` | REGISTER_ONLY anchor files + dedicated index |

A new constant `HoodieTimeline.REGISTER_ONLY_BOOTSTRAP_INSTANT_TS` (`00000000000003`) is introduced. The dedicated instant lets read and write paths identify REGISTER_ONLY file groups directly from `baseFile.getCommitTime()` — no index lookup needed for mode detection.

### Separate Index

REGISTER_ONLY entries are stored in a **dedicated** HFile index alongside the existing METADATA_ONLY index:

```
.hoodie/.aux/.bootstrap/
├── .partitions/                    # METADATA_ONLY (existing)
├── .fileids/                       # METADATA_ONLY (existing)
├── .register_only_partitions/      # REGISTER_ONLY (new)
└── .register_only_fileids/         # REGISTER_ONLY (new)
```

**Why not a shared index?** HFiles are immutable — any mutation requires a full rewrite. With a shared index, even a single-partition promotion forces a complete rebuild of all METADATA_ONLY + REGISTER_ONLY entries. A separate index scopes mutation cost to REGISTER_ONLY entries only:

| Operation | Shared index | Separate index |
|---|---|---|
| Rollback all REGISTER_ONLY | Rewrite entire HFile | Delete dedicated index + commit |
| Promote a partition | Rewrite entire HFile | Rewrite only REGISTER_ONLY HFile |
| Audit REGISTER_ONLY partitions | Scan entire index | Read dedicated index directly |

`HFileBootstrapIndexReader` (in `o.a.h.common.bootstrap.index.hfile`) is extended to transparently union results from both indexes, so the file system view sees all mappings without any changes.

### Rollback Safety

Anchor files are tracked in commit write stats — standard Hudi rollback deletes them. `dropBootstrapIndexIfNeeded` (in `BaseRollbackActionExecutor`) is made instant-aware:
- Rolling back `00000000000001` → drops METADATA_ONLY index only
- Rolling back `00000000000003` → drops REGISTER_ONLY index only

### Write Path

```
SparkBootstrapCommitActionExecutor.execute()
  ├─ metadataBootstrap()        @ 00000000000001
  │   ├─ skeleton files with meta columns
  │   └─ writes main bootstrap index (HFileBootstrapIndexWriter)
  ├─ fullBootstrap()            @ 00000000000002
  │   └─ bulk-insert rewrite
  └─ registerOnlyBootstrap()    @ 00000000000003   [new]
      ├─ empty anchor files
      ├─ writes dedicated register-only index (HFileBootstrapIndexWriter)
      ├─ updates Hudi Metadata Table (partitions + files)
      └─ commits with HoodieWriteStat per file
```

The Metadata Table update in `registerOnlyBootstrap()` is critical: without it, REGISTER_ONLY partitions are invisible to Hive sync and any read path that discovers partitions via the MDT rather than direct filesystem listing.

### Read Path

1. `HoodieFileGroupReaderBasedFileFormat` detects bootstrap tables via the `isBootstrap` flag propagated from `HoodieHadoopFsRelationFactory`
2. For REGISTER_ONLY files, `baseFile.getCommitTime() == "00000000000003"` identifies the mode — data columns are read from the external source, meta columns are padded with NULL
3. Source file path resolved via `HFileBootstrapIndexReader` (unions main + register-only index)

### Write Guardrails

A new `rejectWritesToRegisterOnlyPartitions()` check is added to `BaseSparkCommitActionExecutor`. It runs in parallel via Spark: checks if any base file in targeted partitions has `commitTime == "00000000000003"`. Rejects writes with a clear error listing violating partitions.

### Promotion (REGISTER_ONLY → FULL_RECORD)

```
1. Identify target partitions from REGISTER_ONLY index
2. Read source data via bootstrap file mapping
3. Issue replace-commit creating new FULL_RECORD base files
4. Rewrite REGISTER_ONLY index excluding promoted partitions
5. Hudi cleaner removes old anchor files
```

## Limitations

- REGISTER_ONLY partitions have NULL meta columns — no incremental queries or record-level ops without promotion
- Per-partition promotion rewrites the REGISTER_ONLY index HFile
- COW only in initial implementation
