## Abstract

This RFC proposes native vector similarity search support in Apache Hudi, enabling
approximate nearest neighbor (ANN) queries on embedding columns stored in Hudi tables.
The design extends the new `VECTOR` type in the main data table and stores the vector
index entirely in the Hudi metadata table (MDT). The base table remains the source of
truth for exact vector values, while MDT stores compact cluster-routing, posting, and
RaBitQ metadata for candidate discovery and pruning.

The approach draws on two key research contributions:

- **SPFresh** (SOSP '23): the LIRE protocol for lightweight, incremental cluster
  rebalancing with a provable Nearest-Partition-Assignment (NPA) invariant.
- **RaBitQ** (Johnson-Lindenstrauss): binary quantization via a fixed random rotation —
  no codebook training, no drift, 95%+ recall@100 with exact re-ranking.

## 1. Architecture Overview

### 1.1 Key Architectural Principle: Separation of Data and Routing

```
┌─────────────────────────────┐     ┌──────────────────────────────────────┐
│     MAIN TABLE (Parquet)    │     │ METADATA TABLE (vector_index_<name>) │
│                             │     │                                      │
│  • Full vector data         │     │  • Active generation manifest        │
│  • VECTOR(D) column         │     │  • Cluster centroids (K x D)         │
│  • Standard Hudi lifecycle  │     │  • Sharded posting rows:            │
│  • No vector-index columns  │     │    record_key + cluster/shard + FG  │
│  • Source of truth for      │     │  • Cluster manifests:               │
│    exact re-ranking         │     │    cluster_id -> shards + FGs       │
│                             │     │  • RaBitQ binary codes              │
│                             │     │    embedded in postings             │
│                             │     │  • RaBitQ metadata: seed, dimension, │
│                             │     │    normalization settings            │
└─────────────────────────────┘     └──────────────────────────────────────┘

         ▲                                        ▲
         │                                        │
         │  Exact vectors read                    │  Routing and posting
         │  after candidate                       │  rows read first to
         │  discovery                             │  prune candidates
         │                                        │
         └────────────┐          ┌────────────────┘
                      │          │
                ┌─────┴──────────┴──────┐
                │    Vector Query Plan   │
                │                        │
                │  1. Read manifest      │
                │     and centroids      │
                │  2. Find top-P clusters│
                │     (linear or HNSW)   │
                │  3. Read MDT postings  │
                │     for those clusters │
                │  4. Score binary codes │
                │     with popcount      │
                │  5. Read base vectors  │
                │     for top candidates │
                │  6. Re-rank exactly    │
                │     and return top-K   │
                └────────────────────────┘
```

This separation means:
- **Storage overhead** for the MDT index is O(K x D + N) and is dominated by centroid
  records and one compact posting row per indexed vector. There is no separate full-size
  assignment row family.
- **The base table schema does not change**: cluster placement, file-group location, and
  RaBitQ codes live in MDT postings, not in hidden or generated columns in the main Parquet files.
- **Read-time pruning is MDT-driven**: the query reads manifests, centroids, file-group
  mappings, and sharded posting rows from MDT before touching base-table files.
- **Compaction is incremental**: LIRE-style maintenance updates MDT generations,
  cluster manifests, postings, and file-group mappings without rewriting base-table
  vectors solely to refresh index metadata.
- **No pre-training**: RaBitQ's rotation matrix R is derived from a fixed random seed
  and never expires. Unlike PQ codebooks, it requires no retraining as data drifts.

## 2. Storage Architecture

This design uses the Hudi metadata table as the storage layer for vector-index metadata.
The base table remains the source of truth for the original `VECTOR` column; the metadata
partition stores only the information needed to route, prune, approximately score, and
maintain the index.

The key change from earlier drafts is that RaBitQ payloads are stored in the MDT vector
index partition, not as hidden columns in the base table.

### 2.1 Vector Column in Main Table

Vectors are stored in the base table using the `VECTOR` logical type introduced by RFC-99.
The base table keeps the full vector values and normal user columns:

```text
main table parquet/log files
├── _hoodie_* metadata fields
├── record key fields
├── partition fields
├── user columns
└── embedding VECTOR(D)
```

The vector index does not add `_hudi_vec_*` hidden columns to the base table. This keeps the
base schema stable and avoids coupling ANN index maintenance to base-file rewrites. Exact
re-ranking still reads the authoritative vector values from the base table, but only for the
bounded candidate set produced by MDT posting scans.

#### 2.1.1 Example Base Table Layout

Consider a product table with a four-dimensional embedding column. Production embeddings are
usually much larger (for example, `VECTOR(768)` or `VECTOR(1536)`), but `VECTOR(4)` keeps the
storage shape readable:

```sql
CREATE TABLE products (
  product_id STRING,
  category STRING,
  name STRING,
  embedding VECTOR(4)
) USING HUDI
PARTITIONED BY (category);
```

Example rows:

| product_id | category | name | embedding |
|---|---|---|---|
| `p001` | `electronics` | `wireless mouse` | `[0.12, -0.08, 0.44, 0.89]` |
| `p002` | `electronics` | `usb keyboard` | `[0.10, -0.05, 0.40, 0.91]` |
| `p003` | `books` | `vector search handbook` | `[-0.31, 0.72, 0.18, -0.22]` |

In the base table, the vector is stored with the rest of the record:

```text
products/category=electronics/<file-group>.parquet
├── _hoodie_record_key        = p001
├── _hoodie_partition_path    = category=electronics
├── product_id                = p001
├── category                  = electronics
├── name                      = wireless mouse
└── embedding VECTOR(4)       = [0.12, -0.08, 0.44, 0.89]
```

No vector-index-specific columns are added to this parquet file. The same layout applies to
a high-dimensional column such as `VECTOR(768)`; only the vector payload length changes.

### 2.2 Metadata Table Partition Model

Each vector index is stored as a dynamic metadata partition. The partition name follows the
existing index naming pattern:

```text
.hoodie/metadata/
└── vector_index_<index_name>/
```

For example, `CREATE INDEX embedding_idx ON products USING VECTOR (embedding)` creates the
metadata partition:

```text
.hoodie/metadata/vector_index_embedding_idx/
```

This is one physical MDT partition per vector index. The partition contains multiple logical
row families distinguished by record-key prefixes; it is not modeled as separate subfolders
such as `centroids/`, `assignments/`, or `rabitq_codes/`. Earlier drafts included a separate
assignment family, but the accepted design avoids that duplicate N-row structure. Keeping all row families in one MDT
partition lets Hudi use the existing metadata-table file groups, log files, compaction, and
record lookup/prefix lookup APIs.

Using the example above, an index on `embedding`:

```sql
CREATE INDEX embedding_idx
ON products
USING VECTOR (embedding)
OPTIONS (
  'vector.dimension' = '4',
  'vector.metric' = 'cosine',
  'vector.quantizer' = 'IVF_RABITQ',
  'vector.num_clusters' = '2'
);
```

creates metadata rows under:

```text
.hoodie/metadata/vector_index_embedding_idx/
├── __centroids__
├── __quantizer__
├── __manifest__
├── M|0000007B
├── C|0000007B|00000000
├── C|0000007B|00000001
├── P|0000007B|00000000|0000|p001
├── P|0000007B|00000000|0000|p002
└── P|0000007B|00000001|0000|p003
```

The tree above shows MDT record keys only. The full vector is not stored in the MDT key or
payload; it remains in the base table:

```text
products/category=electronics/<file-group>.parquet
└── p001.embedding VECTOR(4) = [0.12, -0.08, 0.44, 0.89]
```

The MDT posting row stores the compact RaBitQ code derived from that vector:

```text
key:   P|0000007B|00000000|0000|p001
value: HoodieVectorIndexInfo {
         entryType     = POSTING
         generationId  = 123
         clusterId     = 0
         shardId       = 0
         fileGroupId   = <file-group>
         partitionPath = category=electronics
         binaryCode    = 0b00000101   # derived from p001.embedding by RaBitQ
         scalar        = 1.00         # optional, omitted when assumeNormalized=true
       }
```

The exact cluster ids are produced by KMeans during bootstrap. This example assumes `p001`
and `p002` are assigned to cluster `0`, while `p003` is assigned to cluster `1`.

The example keys mean:

| Example key | Meaning |
|---|---|
| `M|0000007B` | Generation manifest for generation `0x0000007B` (`123` decimal). It records generation-level metadata such as quantizer type, code width, RaBitQ seed, and publish time. |
| `C|0000007B|00000000` | Cluster manifest for cluster `0` in generation `123`. It stores cluster-level summary such as shard count, vector count, and candidate file groups. |
| `C|0000007B|00000001` | Cluster manifest for cluster `1` in generation `123`. |
| `P|0000007B|00000000|0000|p001` | Posting row for `p001`: generation `123`, cluster `0`, shard `0`. This stores the packed RaBitQ `binaryCode`, optional `scalar`, file group, and partition path. The posting row is also the canonical record-to-cluster/location mapping. |
| `P|0000007B|00000000|0000|p002` | Posting row for `p002`: generation `123`, cluster `0`, shard `0`. |
| `P|0000007B|00000001|0000|p003` | Posting row for `p003`: generation `123`, cluster `1`, shard `0`. |

The fixed-width hexadecimal fields keep lexicographic ordering aligned with numeric order.
That matters for prefix scans. For example, a query that probes cluster `0` in generation
`123` can scan the prefix `P|0000007B|00000000|` to read all posting shards for that cluster,
or the narrower prefix `P|0000007B|00000000|0000|` to read only shard `0`.

#### 2.2.1 What is a Posting Shard?

An IVF cluster can contain millions of vectors. If all posting rows for that cluster used a
single key range, one hot cluster would create a very large MDT prefix scan and limit
parallelism. A **posting shard** splits one cluster's posting rows into smaller, stable
sub-ranges:

```text
cluster 10
├── shard 0000: P|0000007B|0000000A|0000|*
├── shard 0001: P|0000007B|0000000A|0001|*
├── shard 0002: P|0000007B|0000000A|0002|*
└── shard 0003: P|0000007B|0000000A|0003|*
```

The `C|<generation>|<cluster>` manifest stores `shardCount`. During bootstrap or write-time
indexing, each record is assigned to one shard within its cluster, typically by a stable
hash of the record key:

```text
shardId = hash(record_key) % shardCount
```

Sharding does not change vector semantics. It only controls how posting rows are physically
grouped for MDT prefix scans. Query execution probes clusters first, then scans the selected
cluster's posting shards in parallel.

### 2.3 Logical Row Families

The vector metadata partition stores these logical row families:

| Key family | Example key | Purpose |
|---|---|---|
| `__centroids__` | `__centroids__` | Serialized IVF centroid matrix for the index. |
| `__quantizer__` | `__quantizer__` | Quantizer metadata: type, code width, RaBitQ seed, normalization flag. |
| `__manifest__` | `__manifest__` | Pointer to the active generation used by readers. |
| `M|<generation>` | `M|0000007B` | Immutable generation manifest. |
| `C|<generation>|<cluster>` | `C|0000007B|0000000A` | Cluster manifest: shard count, file groups, vector count. |
| `P|<generation>|<cluster>|<shard>|<record_key>` | `P|0000007B|0000000A|0003|product-123` | Canonical record placement plus RaBitQ posting payload for ANN scoring. |
| `__fg__/<cluster>/<partition_path>` | `__fg__/10/country=US` | Cluster-to-file-group forward map used by Spark pruning and base-table candidate reads. |

The first implementation already includes the key builders and raw-key wrappers for these
families in `HoodieTableMetadataUtil`, `VectorGenerationManifestRawKey`,
`VectorClusterRawKey`, and `VectorPostingPrefixRawKey`.

### 2.4 Index-Level Records

Index-level records are small and are read at query planning or generation activation time.

#### 2.4.1 Centroids Record

`__centroids__` stores the serialized centroid matrix:

```text
key:   __centroids__
value: HoodieVectorIndexInfo {
         entryType     = CENTROIDS
         centroidBytes = serialized K x D centroid matrix
       }
```

Centroids are serialized using the vector element type and dimension from the source
`VECTOR` schema. The Spark POC writes this from
`SparkHoodieBackedTableMetadataWriter#getVectorIndexRecords` using
`VectorIndexBootstrapUtils.serializeCentroids(...)`.

#### 2.4.2 Quantizer Record

`__quantizer__` stores the configuration required to encode and score RaBitQ postings:

```text
key:   __quantizer__
value: HoodieVectorIndexInfo {
         entryType          = QUANTIZER
         quantizerType      = IVF_RABITQ
         quantizedCodeBytes = ceil(dimension / 8)
         randomSeed         = <seed used to regenerate rotation matrix>
         assumeNormalized   = true | false
       }
```

The rotation matrix itself is not stored. `RaBitQEncoder` deterministically regenerates it
from `(dimension, randomSeed)`, which avoids codebook drift and removes any retraining
requirement for the quantizer.

#### 2.4.3 Active Manifest Record

`__manifest__` points readers to one active posting generation:

```text
key:   __manifest__
value: HoodieVectorIndexInfo {
         entryType          = MANIFEST
         generationId       = <active generation>
         quantizerType      = IVF_RABITQ
         quantizedCodeBytes = <code bytes>
         randomSeed         = <seed>
         assumeNormalized   = true | false
         lastUpdatedTs      = <publish time>
       }
```

Readers must use one active generation consistently. This prevents a query from mixing
cluster rows from one rebuild with posting rows from another rebuild.

### 2.5 Generation-Scoped Records

A vector index generation is an immutable set of cluster and posting rows. Rebuild or
rebalancing can create a new generation and publish it by updating `__manifest__`.

#### 2.5.1 Generation Manifest

`M|<generation>` records generation-level metadata. It is useful for validation, debugging,
and cleaner decisions:

```text
key:   M|0000007B
value: HoodieVectorIndexInfo {
         entryType          = MANIFEST
         generationId       = 123
         quantizerType      = IVF_RABITQ
         quantizedCodeBytes = 96
         randomSeed         = 42
         assumeNormalized   = false
         lastUpdatedTs      = <build time>
       }
```

#### 2.5.2 Cluster Manifest Rows

`C|<generation>|<cluster>` rows summarize each IVF cluster:

```text
key:   C|0000007B|0000000A
value: HoodieVectorIndexInfo {
         entryType     = CLUSTER
         generationId  = 123
         clusterId     = 10
         shardCount    = 4
         fileGroupIds  = [fg1, fg7, fg9]
         vectorCount   = 125000
         lastUpdatedTs = <build time>
       }
```

The read path uses these rows to determine which posting-shard prefixes to scan and which
base-table file groups remain candidates after centroid probing.

### 2.6 No Separate Assignment Rows

RFC-104 intentionally does **not** maintain a separate `A|<record_key>` assignment row
family. Early drafts included one, but testing showed that a full N-row assignment family
would duplicate state already present in postings and materially increase MDT size and
write amplification.

Instead, each posting row is the canonical assignment record for that vector. The posting
key identifies the active generation, IVF cluster, posting shard, and record key, while
the payload carries the base-table file-group and partition pointer:

```text
key:   P|0000007B|0000000A|0003|product-123
value: HoodieVectorIndexInfo {
         entryType     = POSTING
         generationId  = 123
         recordKey     = product-123
         clusterId     = 10
         shardId       = 3
         fileGroupId   = fg7
         partitionPath = country=US/category=electronics
         binaryCode    = <packed RaBitQ bits>
         scalar        = <norm, null when assumeNormalized=true>
       }
```

This mirrors Hudi's record-level index principle: store the record-to-location mapping
once, in the structure already required by the read path. Maintenance jobs derive cluster
placement by scanning the affected posting ranges instead of maintaining a second N-row
assignment index.

### 2.7 Posting Rows

`P|<generation>|<cluster>|<shard>|<record_key>` rows store the compact RaBitQ payload used
for approximate scoring:

```text
key:   P|0000007B|0000000A|0003|product-123
value: HoodieVectorIndexInfo {
         entryType     = POSTING
         generationId  = 123
         clusterId     = 10
         shardId       = 3
         fileGroupId   = fg7
         partitionPath = country=US/category=electronics
         binaryCode    = <packed RaBitQ bits>
         scalar        = <norm, null when assumeNormalized=true>
         lastUpdatedTs = <write/build time>
       }
```

The key order is intentional:

```text
P|generation|cluster|shard|record_key
```

It supports prefix scans over selected cluster shards:

```text
P|0000007B|0000000A|0003|
```

This is the MDT-native replacement for hidden-column RaBitQ storage and for a separate
assignment family. For `D=768`, each posting stores a 96-byte binary code plus an optional
scalar and small routing metadata.

### 2.8 File-Group Mapping Rows

File-group mapping rows are transitional compatibility records. The preferred design stores
cluster-to-file-group summaries in `C|<generation>|<cluster>` manifests so the query planner
can prune base-table file groups without scanning posting rows first:

```text
key:   __fg__/<cluster>/<partition_path>
value: HoodieVectorIndexInfo {
         entryType     = FG_MAPPING
         clusterId     = 10
         partitionPath = country=US/category=electronics
         fileGroupIds  = [fg1, fg7, fg9]
         vectorCount   = 125000
         lastUpdatedTs = <update time>
       }
```

The row is keyed by `(cluster, partition_path)` because vector clusters can span multiple
Hudi partitions. Partition-scoped mapping lets a query with a partition predicate, such as
`category = 'electronics'`, avoid opening file groups from unrelated partitions.

For example, if the query probes cluster `10` and Spark has already pruned the base table to
`category=electronics`, `VectorIndexSupport` only needs the row
either the `C|<generation>|10` manifest or, while compatibility rows exist,
`__fg__/10/category=electronics` to identify candidate file groups `[fg1, fg7, fg9]`.

### 2.9 Metadata Table File-Group Partitioning

The vector index partition can become large because posting rows scale with vector count.
The row-key design should allow the MDT partition to distribute data across metadata table
file groups while preserving useful prefix locality.

Design principles:

1. **One dynamic MDT partition per vector index.** Each vector column/index is independently
   registered, built, cleaned, and queried.
2. **Prefix locality for query scans.** Posting keys group rows by generation, cluster, and
   shard so selected cluster shards can be read by prefix.
3. **Shard large clusters.** `shardId` prevents a single hot cluster from creating one very
   large prefix range. The current POC derives shard count from cluster population and maps
   records to shards with a stable record-key hash.
4. **Small metadata rows stay point-readable.** `__manifest__`, `__quantizer__`,
   `__centroids__`, and `C|...` rows are small and should be loaded in-memory by the query
   planner.
5. **No base-table hidden columns.** All ANN-specific payloads live in MDT row families.

### 2.10 Storage and Read Characteristics

For `N` vectors, dimension `D`, and `K` clusters:

| Row family | Cardinality | Hot path? | Notes |
|---|---:|---|---|
| `__centroids__` | 1 | Yes | Loaded for centroid probing. |
| `__quantizer__` | 1 | Yes | Provides RaBitQ seed and code width. |
| `__manifest__` | 1 | Yes | Selects active generation. |
| `M|...` | generations | No | Lifecycle/validation metadata. |
| `C|...` | `K` per generation | Yes | Cluster-level shard and file-group summary. |
| `P|...` | `N` per generation | Yes, prefix-scanned | Canonical assignment/location plus approximate scoring payload. |
| `__fg__/...` | `K * partitions` | Transitional | Current Spark pruning compatibility. |

The query hot path reads small index-level metadata, probes a small number of clusters, and
prefix-scans only the selected posting shards. It does not maintain or scan a duplicate assignment family and does not duplicate full
vectors into MDT.

## 3. Index Algorithm: IVF + RaBitQ

The first vector index algorithm is intentionally simple: IVF for coarse routing and
RaBitQ for compact approximate scoring.

At a high level:

1. IVF groups vectors into clusters using centroids.
2. A query vector is compared with centroids to select the closest clusters.
3. Hudi uses the MDT file-group mapping to prune base-table file groups.
4. Hudi scans MDT posting rows for the selected clusters.
5. RaBitQ scores those compact postings and returns a shortlist.
6. Hudi reads exact vectors from the base table only for the shortlist and re-ranks them.

### 3.1 Why RaBitQ

RaBitQ is a good fit for Hudi because it avoids a trained product-quantization codebook.
Instead, it uses a deterministic random rotation derived from a stored seed. That means the
quantizer metadata is small and stable:

```text
quantizerType      = IVF_RABITQ
randomSeed         = 42
quantizedCodeBytes = ceil(dimension / 8)
assumeNormalized   = false
```

For a `VECTOR(768)` column, each RaBitQ code is `768 / 8 = 96` bytes. For the smaller
`VECTOR(4)` example in Section 2, the code is one byte because four sign bits fit into a
single byte.

### 3.2 Encoding a Vector

For each vector `v`, RaBitQ stores a compact binary representation in the MDT posting row:

```text
1. norm = ||v||
2. normalized = v / norm
3. rotated = R @ normalized       (R is regenerated from the stored randomSeed)
4. binaryCode = sign(rotated)     (one bit per dimension)
5. scalar = norm                  (omitted when assumeNormalized=true)
```

Example with `VECTOR(4)`:

```text
v         = [0.12, -0.08, 0.44, 0.89]
rotated   = [0.51, -0.22, 0.03, -0.61]   # illustrative
sign bits = [1, 0, 1, 0]
binaryCode = 0b00000101
scalar     = ||v||
```

The MDT posting row stores this compact payload:

```text
P|0000007B|00000000|0000|p001
  binaryCode = 0b00000101
  scalar     = 1.00
```

The exact vector remains only in the base table:

```text
embedding VECTOR(4) = [0.12, -0.08, 0.44, 0.89]
```

### 3.3 Scoring a Query

For a query vector `q`, the read path applies the same transformation:

```text
1. normalize q
2. rotate q using the same R
3. convert sign(rotated_q) to queryBinaryCode
4. compare queryBinaryCode with posting binaryCode using Hamming distance
5. convert the Hamming distance into an approximate vector distance
```

Example:

```text
q               = [0.11, -0.02, 0.39, 0.88]
normalize(q)    = [0.11, -0.02, 0.40, 0.91]
R               = [[ 0.50,  0.50,  0.50,  0.50],   # illustrative; generated from randomSeed
                   [ 0.50, -0.50,  0.50, -0.50],
                   [ 0.50,  0.50, -0.50, -0.50],
                   [ 0.50, -0.50, -0.50,  0.50]]
rotated_q       = R @ normalize(q)
                = [0.70, -0.21, -0.61, -0.10]
sign(rotated_q) = [1, 0, 0, 0]              # positive => 1, non-positive => 0
queryBinaryCode = 0b00000001                # bit 0 is dimension 0, bit 1 is dimension 1, ...
postingCode     = 0b00000101
xor             = 0b00000100
hammingDistance = 1
```

`normalize(q)` means scaling the query vector to unit length:

```text
normalize(q) = q / ||q||
```

For the example above, `q = [0.11, -0.02, 0.39, 0.88]`:

```text
||q|| = sqrt(0.11^2 + (-0.02)^2 + 0.39^2 + 0.88^2)
      = sqrt(0.0121 + 0.0004 + 0.1521 + 0.7744)
      = sqrt(0.9390)
      ≈ 0.969

normalize(q) = [0.11 / 0.969, -0.02 / 0.969, 0.39 / 0.969, 0.88 / 0.969]
             ≈ [0.1135, -0.0206, 0.4025, 0.9082]
             ≈ [0.11, -0.02, 0.40, 0.91]   # rounded to two decimals
```

Here, "rotation" means multiplying that normalized vector by a deterministic random
orthogonal matrix `R`. The example uses a 4x4 matrix only because the example vector is
`VECTOR(4)`. In general, `R` is `D x D`, where `D` is the vector dimension. For a
`VECTOR(768)` column, RaBitQ uses a 768x768 rotation matrix. This changes the coordinate
basis before taking signs, but preserves the vector's similarity structure well enough for
binary approximation. The matrix is not stored per row; it is regenerated from the RaBitQ
`randomSeed` and vector dimension.

The same `R` must be used for both write-time postings and query-time encoding. If postings
were encoded with one rotation and the query were encoded with another, the sign bits would
not be comparable.

The binary code is created by packing the sign bits into a byte. In this example,
`sign(rotated_q) = [1, 0, 0, 0]`, so only dimension `0` is positive. Dimension `0` maps to
bit `0`, producing `00000001`. If dimensions `0`, `1`, and `2` were positive, the packed
code would be `00000111`.

A lower Hamming distance means the posting is a better approximate match. Hudi keeps the
best candidates from the selected posting rows, then performs exact re-ranking with the
original `VECTOR` values from the base table.

### 3.4 What a Posting Row Stores

A posting row is one MDT record for one indexed base-table row. Its record key identifies
where the posting lives in the ANN structure:

```text
P|<generation>|<cluster>|<shard>|<record_key>
```

Its payload stores only the metadata needed to score and locate that base-table row:

| Posting payload field | Purpose |
|---|---|
| `binaryCode` | RaBitQ sign bits used for approximate Hamming-distance scoring |
| `scalar` | Optional normalization/scaling value used by RaBitQ when vectors are not assumed normalized |
| `generationId` | Index generation that produced this posting |
| `clusterId` and `shardId` | IVF cluster and posting shard used for lookup and parallel scans |
| `fileGroupId` and `partitionPath` | Pointer back to the base-table file group for final exact re-ranking |

The posting row does not store the original vector or any other base-table columns. Those
values remain in the base table. The metadata table only stores enough compact state to find
candidate rows and then read the original `VECTOR` values for exact scoring.

## 5. Query Execution Flow

### 5.1 Query Cardinality

The read path supports both single-query and batched-query ANN. A single query vector is
only the simplest case:

```sql
SELECT id, name
FROM products
WHERE approx_nearest_neighbors(embedding, ARRAY(0.1, 0.2, ...), 10)
```

At Hudi scale, engines must also support a relation of query vectors:

```sql
SELECT q.query_id, p.id, vector_distance(q.embedding, p.embedding) AS distance
FROM query_vectors q
JOIN products p
  ON approx_nearest_neighbors(p.embedding, q.embedding, 10)
```

The second form may contain thousands or hundreds of thousands of query vectors. The
implementation must not run the single-vector metadata path independently for every query
row. Instead, it batches query vectors, shares MDT reads, and keeps independent top-K state
per `query_id`.

### 5.2 Batched MDT Query Plan

For a batch of query vectors `Q = {q1, q2, ... qN}`, the read path is:

```text
1. Resolve the active vector-index generation from __manifest__.
2. Load generation metadata, centroids, and RaBitQ quantizer metadata once.
3. Apply normal Hudi partition pruning and non-vector predicates.
4. For each query vector:
   a. normalize and RaBitQ-encode the query vector
   b. probe centroids and select top-P clusters
5. Group query vectors by selected cluster and posting shard.
6. Scan each selected MDT posting range once:
   P|<generation>|<cluster>|<shard>|*
7. Score posting binaryCode values against all query vectors assigned to that range.
8. Maintain top-R approximate candidates per query_id.
9. Read the union of candidate base-table rows and their original VECTOR values.
10. Re-rank exactly per query_id and return top-K per query vector.
```

This changes the physical shape from "one query vector scans metadata" to "many query
vectors share metadata and base-table reads":

```text
query_vectors
  -> encode queries and probe centroids
  -> explode (query_id, cluster_id, shard_id)
  -> group by (cluster_id, shard_id)
  -> scan MDT P|... posting ranges
  -> score postings and keep top-R per query_id
  -> read candidate base rows once
  -> exact re-rank per query_id
```

The base table may contain PBs of data and billions of vectors. The query batch size is a
separate execution concern; Spark and Flink can scale it with partitioning, shuffle, state,
and memory limits.

### 5.3 Single-Vector Flow

For `N = 1`, the batched plan reduces to the familiar single-vector flow:

```text
query vector
  -> top-P centroid probes
  -> selected MDT posting ranges
  -> RaBitQ approximate top-R candidates
  -> base-table VECTOR read
  -> exact top-K
```
